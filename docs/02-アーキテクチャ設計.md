# Netero — アーキテクチャ設計

> 設計ドキュメント 02 / 前提: [00 結論と戦略サマリ](00-結論と戦略サマリ.md) / [01 プロダクト設計](01-プロダクト設計.md)

## 結論: モバイルファースト・local-first + 薄いソーシャルサーバ

```
┌─ 一般ユーザ（主役）────────────────┐      ┌─ Netero サーバ ─────────────────────┐
│ モバイルアプリ（PWA → Expo）        │      │  Hono API + Postgres                 │
│  local-first キャッシュ + 同期キュー │─────▶│  ├ 認証 (OAuth: GitHub/Google)       │
│  （IndexedDB/SQLite）              │ タスク │  ├ リーグ編成・ランキング (cron)       │
│  1タップ done・即利用開始           │ 同期  │  ├ Live フィード (SSE)               │
└────────────────────────────────────┘      │  └ X 表彰 bot (cron, opt-in)          │
┌─ パワーユーザ（コネクタ）───────────┐      │                                      │
│ 手元のタスク管理ツール（ローカル）   │─────▶│  ※ タスクは公開コンテンツとして        │
│  + sync-client（小さな連携）        │ HTTPS │    サーバ保持（タスクSNS）             │
└────────────────────────────────────┘      └──────────────────────────────────────┘
```

### クライアント戦略: モバイルファースト・local-first

- **PC 常駐ツールは導入障壁として致命的**。ターゲット層（積み上げ勢）の生活動線はスマホにあり、done は日中の生活の中で発生する。記録デバイスはスマホ一択。
- **中身は軽量クライアント**に絞る（1タップ done + タスクリスト + 草グラフ + リーグ + フィード）。多機能な個人タスク管理ツールを丸ごと移植はしない。
- **local-first**: タスクは公開コンテンツとしてサーバに同期。端末内（IndexedDB / SQLite）は即時 UX とオフライン記録のためのローカルキャッシュ + 書き込みキュー。オフラインで done を押しても復帰時に同期され streak が切れない。
- **匿名ファーストの onboarding**: インストール（or URL を開く）だけでアカウントなしで即 +1。リーグ/ランキング参加時に初めて OAuth。導入ハードルを「開く → 押す」まで下げる。
- **配布は PWA → Expo の2段階**: まず PWA（審査なし・ホーム画面追加可）で出し、検証通過後に Expo(React Native) でストア配布（継続系はホーム画面常駐と push 通知が効く）。

### PWA のデータ管理

- **保存先は IndexedDB**（ブラウザ内蔵オブジェクト DB。オフライン動作・PWA の標準解）。生 API は煩雑なので **Dexie.js** を1枚かませる。
- **ストレージ抽象を最初から切る**: `ActivityRepository` 的なインターフェースの背後に IndexedDB 実装を置き、Expo 移行時は SQLite 実装（expo-sqlite）に差し替えるだけにする。
- **消失リスク対策**（IndexedDB はブラウザ都合で消され得る。特に iOS Safari は非インストール Web データを7日で削除）: ① `navigator.storage.persist()` 要求 ② ホーム画面インストール誘導 ③ **サーバ同期が耐久性バックアップを兼ねる**（端末ロストでも復元可能）④ JSON export。
- **streak 計算は依存ゼロの純関数モジュール（TS）+ テスト**として実装し、クライアント（IndexedDB 上）とサーバ（Postgres 上）で同じ関数を使い数字の食い違いを防ぐ。ルールは「隣接日継続で +1 / 途切れでリセット / 最終アクションが昨日以前なら表示上 0」。

## サーバ側の構成

### スタック

| 層 | 選定 | 理由 |
|---|---|---|
| API | **Hono**（Node or Cloudflare Workers） | ポータブル。最初 Node、伸びたら edge へ |
| DB | **Postgres + Drizzle ORM** | userId 分割・トランザクション・集計。マネージド（Neon / Supabase）で運用ゼロ |
| 認証 | **OAuth（GitHub / Google）のみ** | パスワード管理を持たない = 実装とリスク最小。ターゲット層は両方持っている |
| リアルタイム | **SSE** | フィードは片方向配信で足りる。WebSocket 不要（もくもくスプリント v2 で再検討） |
| フロント | React + Vite の **PWA**（Phase 3 で Expo/React Native へ） | モバイルファースト・local-first |
| ホスティング | マネージド一択（Fly.io / Railway / Cloudflare + Neon） | インフラ作業を発生させない |
| AI | **なし**（MVP に AI 機能を入れない） | マルチテナント AI の複雑さを回避 |

### データモデル（最小）

```
users        (id, handle, avatar, oauth_provider, oauth_id, created_at)
arenas       (id, slug, title, status: open|waiting, created_at)   -- テーマアリーナ
tasks        (id, user_id, arena_id, title, body, status: doing|done,
              created_at, completed_at)   -- 公開コンテンツ（タスクSNS）
activities   (id, user_id, task_id, date, kind: create|push|done, created_at)
streaks      (user_id, current, longest, last_action_date)   -- activities から導出・キャッシュ
leagues      (id, arena_id, week_start, tier)                -- リーグはアリーナ内で編成
league_members (league_id, user_id)
reactions    (id, to_user_id, from_user_id, date)            -- 🔥 のみ
```

### リーグ編成・集計

- 毎週月曜 0:00 に cron でリーグ再編成（直近4週の週平均 done 数でタイア分け → 5〜10人単位）。
- コミュニティ運営の人的コストを**ゼロにする**（手動マッチング・手動モデレーションを要する設計は却下）。

## プライバシー・治安設計

- **タスクは公開が前提**（タスクSNS）。「X に書くのと同じ覚悟で書く場所」であることをオンボーディングと規約で明示。非公開タスク機能は作らない — 見せたくないタスクはそもそもここに書かない、が正しい使い分け。
- ハンドルは自由入力（実名強制なし）。性別・年齢等の属性欄を作らない。
- **治安（商材屋・スパム）**: ①コメント・DM を作らない（勧誘の会話導線がない）②フィード内 URL を非リンク化 ③通報 → 非表示の最小モデレーション ④「AIで稼ぐ」アリーナは特に商材屋を引き寄せるため β期は招待制で入口を絞る。
- 退会 = users + tasks + activities の物理削除。
