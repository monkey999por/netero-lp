## マネタイズ: 広告か課金か → **freemium 課金。広告はやらない**

| 方式 | 判定 | 理由 |
|---|---|---|
| **広告** | ❌ やらない | 個人開発で広告が意味を持つのは DAU 万単位から。「集中して積み上げる」アプリに広告は体験の自殺 |
| **freemium 課金** | ✅ 主軸 | ソーシャル機能はサーバコストが恒常的にかかる = サブスクと構造が一致。「継続の強制力」に金を払う前例は多数（習慣化アプリのプレミアム、パーソナルジム、コーチング）。寝てても課金が走る |
| **罰金プール手数料** | 🔒 v2 保留 | コミットメントデバイスとして最も動機整合的（払いたくないから続ける）。ただし賭博・資金決済法の**法務確認が通るまで封印**。Beeminder / StickK に前例あり |
| **B2B チーム版** | 🔮 将来枠 | 「チームの稼働の気配」は開発組織にも刺さり得るが B2B 営業はソロの時間を食う。個人版が回ってから |
| **買い切り** | ❌ | サーバコストが恒常なのに収入が一回きり。構造矛盾 |

### 無料/有料の線引き（気配は無料、記憶と装備は有料）

**コア体験（フィード・リーグ・草グラフ）は無料**。気配アプリはユーザ数そのものが商品価値（人が少ない自習室に意味はない）なので、無料側を絞るとプロダクトが死ぬ。

| | Free | Premium（月500円 / 年5,000円） |
|---|---|---|
| 草グラフ・streak | 直近90日 | 全履歴 + 年次レポート |
| リーグ | 自動編成1つ | プライベートリーグ作成・複数所属 |
| フィード | 見るだけ | 自分の done をハイライト |
| ロール（v2） | 割当のみ | ロール選択権 |
| streak 保険 | なし | 月1回の「病欠チケット」（1日欠けても streak 維持）|
| connector API | 手動 +1 のみ | 外部ツールからの自動 sync |

- **streak 保険は実証済みの強い課金動機**（Duolingo の Streak Freeze）。streak が長い人ほど失うのが怖い = 継続している優良ユーザほど払う。
- **connector API（自動 sync）を Premium に置く**ことで、ヘビーユーザ＝開発者層から自然に回収。

### 価格仮説と損益分岐（概算）

- 固定費: マネージド DB + ホスティング + ドメインで **月 $10〜25**（Neon/Fly の無料〜最小枠で開始）。
- 月500円 × 有料転換率 5%（習慣系アプリの相場 3〜8%）:
  - 登録 100人 → 有料5人 → 月2,500円 ≒ 損益分岐
  - 登録 1,000人 → 有料50人 → 月25,000円
  - 登録 4,000人 → 有料200人 → 月10万円
- **含意**: short-term の即金源ではない。ポートフォリオの1枚として、時間をかけて育てる前提。

## マーケティング運用

### 基本方針: build-in-public（使うこと自体が宣伝になる）

最大のマーケ資産は「**自分の積み上げを公開する人たちがターゲット**」という構造。ユーザの共有行動がそのまま獲得チャネルになるループを最初から設計に組み込む:

```
共有画像 export（自分の streak 画像を X に投稿）
  → 画像に Netero のクレジット + LP リンク
  → #今日の積み上げ タグの数千件/日の流れに乗る
  → LP → ウェイトリスト → β招待 → βユーザも共有画像を使う → ループ
```

- **作者が広告塔第1号**（ドッグフーディング）。開発ログも X で公開（build-in-public は開発者向けプロダクトと相性が実証済み）。**このリポジトリを public にしているのもその一環** — 設計を公開し、新アリーナの提案を受け付ける。
- **獲得チャネルの優先順位**: ① X の積み上げ/朝活タグ（無料・ターゲット直撃） ② 既存習慣化アプリの不満層（比較記事 SEO） ③ Product Hunt / 個人開発コミュニティ（β開放時に1回）。
- **やらないマーケ**: 有料広告（単価が合わない）、インフルエンサー依頼、複数 SNS 同時運用（X 1本に絞る）。

### 公式 X アカウント運用: ランキング表彰 bot + 投票

「誰がどれだけタスクをこなしたかを X で発信し、上位者・継続者への投票を持つ」を採用。**ただし投票はアプリ内に作らず X のネイティブ投票（ポール）で行う**:

- **開設は Phase 1 の LP と同時**（フォロワーはフェーズをまたぐ複利資産）。
- **自動投稿の型**（掲載は opt-in ユーザのみ）: 週次のリーグ昇格者・週間 done 数トップ、**改善率トップ**（絶対数トップは同じ顔ぶれ化するので、普通の人が載れる枠を必ず混ぜる）、streak 節目（30/100/365日）の自動祝福。
- **MVP 投票は X ポールで**: 週次で opt-in ノミネート → 投票 → 優勝者を固定ポストで表彰。実装ゼロ・治安リスクをアプリ外に隔離・投票がそのまま公式アカウントのエンゲージメントになる。
- **絶対ルール**: 掲載は opt-in 限定。最下位・停滞者に触れる企画は永久にやらない。

### コミュニティ運用コスト（寝てても回るか）

| 運用タスク | 対策 |
|---|---|
| リーグ編成 | cron 全自動 |
| モデレーション | コメント機能を作らないので炎上面積が構造的にゼロ。リアクションは 🔥 のみ |
| 問い合わせ | FAQ + Issues |
| コンテンツ供給 | ゲーム型を採らないので恒常供給義務なし。週次集計は自動生成 |

→ 定常運用は**週1時間以内**を設計目標とする。

### 検証運用（Phase 1）

**このリポジトリで LP を配信**。テーマ（アリーナ）ごとに独立した LP を1ドメイン・パス違いで並べ（`/ai-earn/` `/toeic/` …）、デザインはマーケごとに1本ずつ自由に変えて色々なパターンを試す。テーマごとの登録数が**そのままアリーナ需要の事前調査**になる。フォームもテーマごとに分ける。

1. 30日間、X の積み上げ投稿に LP リンクを添えるだけ。追加の宣伝作業はしない。
2. 30日後に 00 の撤退ライン（登録30件 / メール記入率50% / 期待度スコア・訴求シグナル）で判定。テーマ別に独立判定し、通ったテーマだけ Phase 2 へ。

## リスク一覧（正直に）

| リスク | 深刻度 | 緩和 |
|---|---|---|
| コールドスタート（人がいないフィードは逆効果） | **高** | β は招待制で密度を確保。全体でなくリーグ内フィードを先に出す |
| ネットワーク効果前提の個人開発は分が悪い | **高** | **5人集まれば1リーグ成立**する最小成立単位の小ささが救い。Phase 1 で需要の事前証拠を取ってから作る |
| 作者の飽き | 中 | フェーズゲートと撤退ラインで「ずるずる作り続ける」を構造的に防止 |
| 大手が同機能を後追い | 中 | 大手は「証明写真・監視」路線に投資済みで方向転換しにくい。「気配」ポジションを先に取る |
